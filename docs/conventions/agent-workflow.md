> 本文档补充 `AGENTS.md`：描述**跨阶段**的做事方式、**验证证据**与常见偷懒的反驳。仓库专属约束（目录、契约、Layout 禁区、文档同步）仍以 `AGENTS.md` 与 `.cursor/rules/` 为准。

## 2. 推荐生命周期（简版）

不必每票都走全量，按风险与体量裁剪；**高风险或大范围改动**应覆盖更多环节。

1. **澄清**：需求是否模糊？模糊则先收敛目标、非目标、验收口径（可书面记在 issue/PR 或 PRD 草稿）。
2. **规格**：是否触及新行为、新 API、新数据？先对齐 PRD / 契约 / 索引中已有条目，再写代码。
3. **拆解**：多文件、多包时拆成**可独立验证**的小步；标明依赖顺序与验收条件。
4. **实现**：最小有效变更；第三方框架/API 行为以**官方文档**为准，不确定处写明假设或加注释链接出处。
5. **验证**：用**可展示证据**证明（见第 4 节），禁止仅以「看起来对」收工。
6. **评审前自检**：安全边界、错误路径、回滚/兼容；必要时做「对抗式」过一遍关键断言（见第 5 节）。
7. **交付**：若涉及发布或迁移，写明回滚点与监控/验证步骤（与 `docs/prd/monorepo.md`、CI 实践一致处从之）。

## 3. 与「读取策略」的配合

`AGENTS.md` 第二节的 L1/L2/L3 解决**读什么**；本节解决**做完什么算数**：

- 读完 L1 仍要动手时，在脑中过一遍：是否已满足第 2 节中当前任务对应的环节。
- **不要**为遵守本文件而通读所有 PRD；用索引按需打开。

---

## 4. 验证与证据（非可选时的底线）

下列情形在声称「完成」前应留有**可复查**证据（命令输出、关键截图、或 PR 描述中的复现步骤）：

- 修改或新增自动化测试所覆盖的逻辑。
- 修改构建、类型检查、CI 配置。
- 修复用户可见 bug 或浏览器专属行为。
- 变更鉴权、多租户边界、持久化格式或迁移。

**建议命令粒度**（按任务选最小子集，与 AGENTS「最小验证」一致）：

- 前端相关：`pnpm --filter @portal/web exec tsc --noEmit` 或包内既有 test/lint 脚本。
- 后端相关：`pnpm --filter @portal/server test` 或等价。
- 全仓大改再考虑 `pnpm -r typecheck`（耗时，非每改必跑）。

证据写在哪里：PR 描述、commit message 脚注、或 issue 评论均可，但要**具体**（例如「已跑 xxx，通过」而非「测过了」）。

## 5. 高风险时的对抗式自检（简版）

在**生产数据、不可逆迁移、安全敏感路径**或陌生模块上工作时，对关键结论快速走一遍：

1. **陈述**：当前方案的关键断言是什么？
2. **依据**：来自代码、官方文档，还是推断？
3. **质疑**：若依据错误，最坏情况是什么？
4. **收口**：能否用更小改动、更强校验或分阶段发布降低风险？

这与「多写代码」无关；目的是减少事后排障成本。

## 6. 常见借口与反驳（反自我合理化）

| 借口 | 反驳 |
|------|------|
| 先合再补测试 | 缺测试的合并把成本推给下一个人；能写测试的改动应同事务内完成。 |
| 文档回来再补 | 见 `30-doc-sync-on-completion`：结构/页面变更与索引不同步会误导后续 Agent 与人。 |
| 官方文档太长不看了 | 对易变行为（框架版本、API 语义）跳过文档 = 高概率返工。 |
| 只改一行不用跑检查 | 一行也可能触发类型或契约连锁失败；至少跑与改动包相关的最小检查。 |
| 布局我凭感觉对齐 | 涉及 `app/[lang]/` 页面与壳层时必须先满足 Layout Decision 与相关 specs。 |

## 7. 红旗（出现则放慢或拆分）

- 单次 PR 混杂多个无关主题或难以一句话说清「为什么」。
- 改 API 但未同步 `packages/shared` 或调用方。
- 改 schema 但未走迁移生成流程（见 `AGENTS.md` 数据库变更）。
- 在禁止清单路径上「顺手重构」（如未获 PRD 明确授权的 `provider/*`、`components/ui/*`）。
- 无法在一两句话内说明如何**验证**本次改动。
